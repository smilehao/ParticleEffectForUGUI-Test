## 1. NanoMonitor 性能监视器

`NanoMonitor` 是一个轻量级的运行时性能监控面板，专为 UIParticle 系统设计，用于实时查看关键性能数据。

### 1.1 启动方式
在 Unity 编辑器中运行包含 `UIParticle` 组件的场景，监视器窗口将自动显示在屏幕**顶部**（默认配置）。

> **提示**：如果未看到窗口，请检查 `Project Settings > UI > Nano Monitor` 确保已启用，或检查场景内是否有激活的 `UIParticle` 组件。

### 1.2 界面字段说明

监视器面板包含以下核心数据：

| 字段 | 含义 | 单位 | 说明 |
| :--- | :--- | :--- | :--- |
| **Screen** | 当前游戏窗口分辨率 | px | 格式：宽 x 高。分辨率越高，填充率压力越大。 |
| **UIParticles** | 场景中激活的 `UIParticle` 组件数量 | 个 | 反映渲染对象的复杂度。数量过多可能增加 DrawCall。 |
| **Particles** | **当前所有粒子系统中存活的粒子总数** | 个 | **核心指标**。直接决定顶点计算量和 GPU 填充压力。<br>🟢 < 500 (优)<br>🟡 500 - 2000 (良)<br>🔴 > 2000 (需优化) |
| **Time** | UIParticle 系统更新耗时 | ms | **CPU 开销**。仅统计网格构建、顶点变换等逻辑耗时，不含渲染时间。<br>建议保持在 1.0ms 以下。 |
| **Materials** | 材质球缓存数量 | 个 | 反映合批潜力。数量越少，越有利于减少 DrawCall。 |

### 1.3 配置位置
- **配置文件**：`Assets/ProjectSettings/NanoMonitor.asset`
- **调整锚点**：修改 `m_Anchor` (0=底部, 1=顶部) 可改变窗口位置。
- **刷新间隔**：默认为 0.5 秒，平衡了实时性与性能开销。

## 2. 性能压力测试

仓库提供了专门的压力测试场景，用于模拟极端情况下的性能表现。

### 2.1 测试场景
- **路径**：`Assets/Demo/Performance Demo/UIParticle_PerformanceDemo.unity`
- **脚本**：`UIParticle_PerformanceDemo.cs`

### 2.2 测试步骤
1. 打开上述场景并运行。
2. 通过界面上的滑块或按钮调整参数：
   - **Copy Count**: 增加粒子发射器的实例数量（建议从 10 逐步增加到 100+）。
   - **Scale**: 调整粒子大小，观察填充率变化。
3. 切换优化开关，对比 `Time` 和 `FPS` 的变化：
   - **Enable/Disable**: 验证渲染开销。
   - **Mesh Sharing**: 开启后应显著降低 `Time` 值（减少重复顶点计算）。
   - **Random Group**: 测试分组对合批的影响。

### 2.3 预期结果
- **开启 Mesh Sharing**：在大量相同特效实例下，`Time` 应大幅下降，帧率更稳定。
- **粒子数激增**：当 `Particles` 计数超过 2000 时，低端设备上可能出现掉帧。

## 3. 关键指标解读

### 3.1 Particles (粒子总数)
这是最重要的性能指标。
- **统计逻辑**：遍历所有激活的 `ParticleSystem`，累加 `particleCount`。
- **影响因素**：
  - `Emission` 模块的发射速率。
  - `Max Particles` 上限设置。
  - 粒子生命周期（Lifetime）。
- **注意**：即使 `UIParticle` 组件被禁用，若底层 `ParticleSystem` 仍在运行，仍可能计入总数（取决于具体实现逻辑，当前版本仅统计激活且发射模块开启的系统）。

### 3.2 Time (CPU 耗时)
- **低值 (< 0.5ms)**：特效非常轻量，无性能担忧。
- **中值 (0.5ms - 2.0ms)**：正常范围，但在移动端需注意总帧预算。
- **高值 (> 3.0ms)**：存在卡顿风险。
  - **原因**：可能是单帧粒子数量过多，或未开启 `Mesh Sharing` 导致重复计算。
  - **对策**：减少同屏粒子数，或合并相同材质的特效。

### 3.3 Materials (材质数量)
- **理想状态**：数量远小于 `UIParticles` 组件数。
- **异常状态**：如果每个 `UIParticle` 都对应一个独立材质，说明合批失败，DrawCall 会爆炸。
- **优化**：确保多个特效共用同一套材质球（使用 Atlas 或相同的 Shader 变体）。

## 4. 优化建议

基于监控数据，推荐以下优化策略：

1. **控制同屏粒子数**
   - 目标：移动端同屏活跃粒子数控制在 **1000** 以内。
   - 方法：降低发射速率，缩短生命周期，或使用对象池管理特效启停。

2. **强制开启 Mesh Sharing**
   - 对于重复出现的特效（如金币飘动、背景雪花），务必在 `UIParticle` 组件中设置 `Mesh Sharing = Auto` 或 `Full`。
   - 这能将多个网格合并为一个，大幅降低 CPU 构建开销。

3. **减少材质变体**
   - 尽量让不同的特效共用材质球。
   - 避免在运行时动态修改 Material 属性导致实例化（使用 `MaterialPropertyBlock` 代替）。

4. **按需渲染**
   - 当特效不在屏幕可视范围内时，暂停 `UIParticle` 组件或底层 `ParticleSystem`。
   - 利用 `CanvasGroup.alpha = 0` 并不会停止粒子计算，必须显式禁用组件。

5. **针对低端机降级**
   - 根据设备性能分级，动态调整 `Max Particles` 上限或关闭复杂特效。

## 附录：常见问题

**Q: 为什么 `Particles` 计数为 0 但能看到特效？**
A: 可能是粒子处于预加载阶段尚未发射，或者统计逻辑过滤了某些非激活状态的组件。请检查 `Emission` 模块是否启用。

**Q: `Time` 很高但 `Particles` 很少？**
A: 可能是单个粒子的网格复杂度太高（如使用了复杂的自定义网格），或者 `Simulation Space` 设置为 `World` 导致每帧都需要重新计算世界坐标变换。建议尝试改为 `Local` 空间。

**Q: 如何在代码中获取当前粒子总数？**
A: 可通过 `UIParticleUpdater.TotalParticleCount` 静态属性直接读取（如果已暴露该接口），或遍历 `FindObjectsOfType<ParticleSystem>()` 自行累加。
